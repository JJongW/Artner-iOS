# workflow.report

## 설명
작업 완료 후 최종 리포트 (AI_REPORT.md)를 생성한다.
사람이 3분 안에 "무엇이 바뀌었고, 왜 바뀌었고, 어떻게 검증했는지" 파악 가능.

## 파라미터
- `task_plan_md` (String, 필수): AI_PLAN.md 경로
- `context_md` (String, 필수): AI_CONTEXT.md 경로
- `checklist_md` (String, 필수): AI_CHECKLIST_WORK.md 경로
- `code_diff` (String, 필수): 코드 변경 내용

## 출력 파일
- `AI_REPORT.md` (프로젝트 루트)

## 절차

### Step 1: 입력 문서 읽기
```
1. AI_PLAN.md → 작업 목표, 범위, 요구사항
2. AI_CONTEXT.md → 아키텍처 컨텍스트, 기존 패턴
3. AI_CHECKLIST_WORK.md → 체크 결과 (통과/실패/수동 확인)
4. code_diff → 실제 변경 내용
```

### Step 2: AI_REPORT.md 작성
`templates/AI_REPORT.md` 템플릿 기반:

```markdown
# AI Report — {작업명}

## 0. 메타
- 작업명: {Plan에서}
- 작업 유형: {신규 화면 / API 연동 / 버그 수정 / ...}
- 작업 날짜: {오늘}
- 작업자: AI (Claude)

## 1. 결론 요약 (5줄 제한)
- 무엇을 만들었나/바꿨나: {1줄}
- 사용자에게 어떤 변화가 있나: {1줄}
- 기존 기능에 영향은 없나: {1줄}
- 가장 중요한 리스크 1개: {1줄}
- 다음 액션 1개: {1줄}

## 2. 변경 범위
### 2.1 변경 파일 리스트
{code_diff에서 추출한 파일 목록, [NEW]/[MOD] 마킹}

### 2.2 변경 내용 상세
{핵심 변경점 3-5개, 코드 스니펫 포함}

## 3. 설계/의도 (왜 이렇게 했는가)
{3-8줄: 아키텍처 선택 근거, 트레이드오프, 대안}

## 4. 동작 플로우 (사용자 관점)
1) 사용자가 {시작점}
2) {행동}
3) {결과 화면/상태}
4) {에러/빈값 시 반응}

## 5. 검증 결과 (AI_CHECKLIST 기반)
{체크리스트 결과 요약: 통과/실패/수동 확인}

## 6. 리스크/부작용 가능성
{식별된 리스크 + 완화책}

## 7. TODO / 다음 작업 제안
- [ ] {후속 작업}

## 8. 참고
{API 스펙, 화면 캡처 포인트 등}
```

## 품질 기준
- 1절 "결론 요약"만 읽으면 전체 감 잡기 가능
- 2절의 파일 리스트가 code_diff와 정확히 일치
- 5절의 검증 결과가 체크리스트와 일치
- 실패 항목은 사유 + 다음 액션이 명시됨
