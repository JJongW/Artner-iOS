# workflow.plan

## 설명
사용자 작업 요청을 분석하여 상세한 Plan 문서 (AI_PLAN.md)를 생성한다.

## 파라미터
- `task_description` (String, 필수): 사용자가 요청한 작업 설명

## 출력 파일
- `AI_PLAN.md` (프로젝트 루트)

## 절차

### Step 1: 작업 분석
사용자 요청에서 다음을 추출:
- 작업 유형 (신규 화면 / API 연동 / 버그 수정 / 리팩토링 / UX 개선 / 기타)
- 영향 받는 레이어 (Presentation / Domain / Data / Coordinator)
- 필요한 ios.* 스킬 목록

### Step 2: 프로젝트 컨텍스트 확인
- `AI_MANUAL.md` 읽기 → 아키텍처 규칙 확인
- 관련 기존 코드 탐색 (Glob/Grep)
- 의존 파일/모듈 파악

### Step 3: AI_PLAN.md 작성
`templates/AI_PLAN.md` 템플릿 기반으로 작성:

```markdown
# Plan Document — {작업명}

## 1. 작업 개요
- 작업명: {task_description에서 추출}
- 작성일: {오늘 날짜}
- 작업자: AI (Claude)

## 2. 목표
- 이 작업을 통해 해결할 문제: {구체적 문제 정의}
- 성공 기준: {검증 가능한 기준}

## 3. 범위 (Scope)
### 포함
- {변경할 파일/모듈 목록}
### 제외
- {명시적으로 건드리지 않을 영역}

## 4. 요구사항
- 기능 요구: {상세}
- 제약 조건: {아키텍처 준수사항}
- 우선순위: {P0/P1/P2}

## 5. 의존 관계
- 연관 파일: {파일 경로 목록}
- 연관 스킬: {ios.* 스킬 목록}

## 6. 구현 계획
### Step-by-step
1. {단계 1: 무엇을 어떤 파일에}
2. {단계 2}
3. ...

## 7. 리스크/대책
- 리스크 1: {발생 조건} → 대책: {완화 방안}

## 8. 다음 단계
1. workflow.context_notes로 컨텍스트 생성
2. workflow.checklist_generate로 체크리스트 생성
3. workflow.self_check (사전) 실행
```

## 품질 기준
- 구현 계획이 다른 사람(또는 AI)이 그대로 따라할 수 있을 만큼 구체적
- 각 단계에서 수정할 파일 경로가 명시됨
- AI_MANUAL.md의 아키텍처 원칙과 충돌 없음

## 체크리스트

- [ ] 구현 단계가 순서대로 실행 가능
